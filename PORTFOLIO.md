## 4. 의사결정 로그 (트레이드오프)

면접에서 "왜 X 대신 Y를 골랐어요?" 물어볼 때 답할 거리들.

### 4.1 구글 OAuth → 앱인토스 로그인 (실제 전환 완료)
- **고민:** 개발 단계는 구글 OAuth로 빠르게, 출시 직전 토스 로그인으로 교체 계획
- **발견:** 토스 인앱 웹뷰는 외부 사이트(accounts.google.com) 페이지 이동을 차단 → 구글 로그인 시 `NSURLErrorDomain -1004`. 심사자도 로그인 불가 = 반려 확실
- **선택지 비교:**
  - A안 `appLogin`(OAuth 정석): mTLS 인증서 + 토스 Client ID 메일 신청(발급 수일) 필요 → D-7 마감 리스크 큼
  - B안 `getAnonymousKey()`: 토스 공식 제공 미니앱 사용자 식별 API, 외부 의존·인증서 불필요, 정책 변경에 안전
- **결정:** B안 채택. `getAnonymousKey().hash` → Supabase Edge Function이 결정적 계정 매핑 후 세션 발급 → 기존 RLS(`auth.uid()`) 구조 무수정 유지
- **얻은 것:** 외부 OAuth 제거로 -1004 해결, 발급 대기 의존성 0, 마감 안전
- **트레이드오프:** anonymousKey를 서버에서 토스에 재검증하진 않음. 친구초대 기반 저위험 앱이라 수용 (완전 서버검증=appLogin+mTLS는 이 앱엔 과함)

### 4.2 모델 선택 — Haiku 4.5 → Sonnet 4.5 (v1.2에서 재상향)
- **v1.0/v1.1 (Haiku 4.5):** CLAUDE.md 원안 Sonnet → Haiku로 다운그레이드. 회당 5원, 비용 효율 좋음
- **드러난 한계:** v1.2에서 "글+사진 통합 입력" 도입하니 입력 복잡도 상승 → Haiku가 ① 글 정서와 사진을 통합 못 잡고 사진만 묘사하거나 ② "마음이 절로 녹네, 너도" 같이 문장이 끊겨 어색해짐
- **결정:** **Sonnet 4.5로 재상향**. 사용자가 본 화면에서 직접 어색함 지적 → 캐릭터가 곧 제품인 앱에서 회당 5원 → 14원의 차이는 정당함
- **결과:** 같은 입력에서 글 정서 우선·사진 디테일·문장 완결 모두 안정. 100명 매일 사용 기준 월 비용 ~15,000원 → ~42,000원
- **배운 것:** "비용 최적화"는 입력 복잡도가 낮을 때만 통한다. 멀티모달 + 페르소나 통제가 들어가면 모델 사이즈가 곧 품질 — 작은 모델로는 프롬프트 엔지니어링도 한계가 있음

### 4.3 Supabase 유지 vs Firebase 전환
- **고민:** 토스 로그인은 Firebase의 `signInWithCustomToken`이 연동 더 깔끔
- **판단:** 이미 만든 5테이블 + RLS 정책 다 버리고 다시 짜는 비용이 큼
- **결정:** Supabase 유지 + Edge Function으로 토스 로그인 처리
- **원칙:** "전환 비용을 무시하지 말라."

### 4.4 AI 직접 호출 vs Supabase Edge Function 프록시 (v1.1에서 전환 완료)
- **고민:** API 키 클라이언트 노출 vs 백엔드 함수 추가 작업
- **v1.0:** 직접 호출로 빠르게 출시 (키는 빌드 번들에 포함, 사용량 모니터링으로 완화)
- **v1.1:** `supabase/functions/claude-proxy`로 전환. ① 키는 서버 secret(`ANTHROPIC_API_KEY`)으로 이동 ② **Supabase JWT 검증**으로 로그인 사용자만 호출 가능 ③ 시스템 프롬프트도 서버로 이동(클라이언트 번들에서 안 보임)
- **호환성:** `generateComment(base64, mediaType, userText)` 시그니처 그대로 유지 → 호출부 코드 무수정
- **얻은 것:** 키 노출 0, 호출 권한 통제, 프롬프트 자체도 자산화. 마지막 보안 부채 해소

### 4.5 게시물 수정 정책: 잠금 / 자유 / 1회 한정
- **고민:** BeReal식 진정성 vs UX 부담
- **결정:** 하루 1회 수정 가능. 그 후 잠금
- **얻은 것:** 오타·실수 대응 가능 + 컨셉(매일의 진짜 기록)도 살림

### 4.8 출시 전략: "완벽한 v1.5" vs "출시한 v1"
- **고민:** 토스 특화 기능(결제·친구 연동·푸시) 미포함 → 챌린지 우승 가능성 낮음. 더 보강 후 출시할까?
- **결정:** **v1.0 무조건 마감 내 출시**. 토스 특화 기능은 v1.1로 분리
- **얻은 것:** 마감 안전 마진 확보, 운영 데이터로 v1.1 우선순위 결정 가능
- **포기한 것:** 챌린지 첫 심사에서 우승 가능성 (출시 후 후속 평가 노림)
- **배운 것:** 1인 개발의 핵심은 "출시 자체가 가장 큰 차별화". 미완성 v1.5보다 완성된 v1이 학습·포폴 가치 모두 큼

### 4.7 프로필 아바타: 직접 업로드 vs 외부 API 랜덤
- **고민:** 프로필 사진 어떻게 줄까. 이미지 업로드는 Storage 비용·UX 부담
- **결정:** **DiceBear API** (무료 SVG 생성, fun-emoji 스타일)
- **얻은 것:** 0원, 코드 5줄, 캐릭터 톤 일관성, 사용자가 "마음에 들 때까지 누르기" 재미
- **포기한 것:** 본인 사진 업로드 옵션 (필요하면 v1.x에서 추가)
- **배운 것:** "꼭 직접 만들어야 할 부분"과 "외부 서비스로 충분한 부분"을 구분하는 게 1인 개발 핵심. SaaS의 모든 기능을 inhouse로 짤 필요 없음

### 4.6 게시물 공유 모델: 1대1 vs 다대다 (정규화)
- **고민:** 한 글이 한 그룹에만 보일까(`posts.group_id` 1개), 여러 그룹에 동시 공유될까
- **결정:** N:M 매핑 테이블 `post_groups` 도입. 글 1개 ↔ 그룹 N개
- **얻은 것:** 사용자가 한 사진을 가족·친구·팀에 동시 공유 가능. UX 자연스러움
- **포기한 것:** 단일 컬럼의 단순함. 매핑 테이블 RLS 정책 추가 작업
- **배운 것:** 처음 스키마 설계할 때 "관계가 1:1로 보여도 향후 N:M될 가능성"을 고려해야 한다. 먹픽처럼 단순한 도구형은 1:1로 충분하지만, 소셜 앱은 N:M이 디폴트

### 4.7 운영자 데이터 접근 권한 정책
- **사실:** Supabase service_role 키는 RLS 우회 → 모든 사용자 데이터 접근 가능
- **고민:** 신고/디버깅엔 필요 vs 프라이버시 침해 위험
- **결정:** "필요 시에만 접근" 자율 규율 + 개인정보처리방침에 명시
- **검토:** E2E 암호화는 친구 공유 컨셉상 과한 비용 → 채택 X
- **약관 반영:** "운영·분쟁 해결 목적으로 관리자가 데이터에 접근할 수 있음" 명시 예정

## 5. 트러블슈팅 케이스 (실력 보여주는 자산)

면접에서 "막혔던 거 어떻게 해결했어요?" 답하기 위한 사례들.

### 5.1 PostgreSQL RLS 무한 재귀 오류 (단일 테이블 자기 참조)
- **증상:** `infinite recursion detected in policy for relation "group_members"`
- **원인:** group_members 정책이 자기 자신을 select 하면서 또 RLS 평가 → 무한 루프
- **해결:** `SECURITY DEFINER` 함수(`is_group_member()`) 만들어 RLS 우회 호출
- **배운 것:** RLS 정책 안에서 같은 테이블을 참조할 때는 우회 함수 필수
- **파일:** `supabase/migrations/0003_fix_recursion.sql`

### 5.1b RLS 무한 재귀 — 두 테이블 상호 참조
- **증상:** `posts` ↔ `post_groups` 정책이 서로를 참조 → `infinite recursion detected in policy for relation "posts"`
- **원인:** posts 정책이 post_groups select, post_groups 정책이 posts select → cycle
- **해결:** 양쪽에서 쓰는 검사 로직을 `SECURITY DEFINER` 헬퍼 함수로 추출 (`post_is_shared_to_my_group`, `is_post_owner`)
- **배운 것:** RLS 재귀는 단일 테이블뿐 아니라 **상호 참조**에서도 발생. 정책 설계 시 의존 그래프를 미리 그려보고 cycle 확인. 의심되면 무조건 헬퍼 함수
- **파일:** `supabase/migrations/0009_fix_post_groups_recursion.sql`

### 5.2 timestamptz 인덱스 불가 오류
- **증상:** `functions in index expression must be marked IMMUTABLE`
- **원인:** `(created_at::date)` 가 timezone 영향 받아 STABLE → 인덱스 불가
- **해결:** `posted_date date` 컬럼 분리 + `current_date` 기본값
- **배운 것:** Postgres 인덱스에 함수 쓸 땐 IMMUTABLE 여부 확인. 컬럼 분리가 더 깔끔

### 5.3 npm create vite가 비어있지 않은 디렉토리 거부
- **증상:** CLAUDE.md 파일이 있는 폴더에 `npm create vite .` 실행 → 취소됨
- **해결:** 임시 폴더에 생성 → `Copy-Item` 으로 이동 → 임시 폴더 삭제
- **포인트:** 환경 제약(파일 보호 등) 만났을 때 우회 경로 찾는 능력

### 5.7 timezone 불일치 — UTC vs KST 9시간 오차
- **증상:** 어제 글 올렸는데 오늘 들어오면 스트릭 메시지나 화면 상태가 이상하게 표시
- **원인:** `new Date().toISOString()` 은 UTC 기준. 한국 사용자가 한국 시간 기준으로 사고하는 것과 9시간 차이. DB의 `current_date` 기본값도 서버 timezone(UTC) 기준이라 자정 즈음 사용 시 날짜 어긋남
- **해결:** 모든 날짜 계산을 `getLocalDate()` 헬퍼로 통일 — `getFullYear/getMonth/getDate` 사용한 로컬 시간 기준 'YYYY-MM-DD'. posts insert 시 `posted_date`를 명시적으로 세팅 (DB default 신뢰 X)
- **배운 것:** "한국 사용자만 쓰는 앱"이라도 timezone 일관성은 필수. ISO 문자열은 늘 UTC라는 점을 잊으면 자정 버그가 숨겨져 있다가 출시 후 터짐
- **파일:** Home.jsx, Feed.jsx 의 `getLocalDate()` 헬퍼

### 5.6 다른 사용자 프로필이 RLS에 막혀 "이름없음"으로 표시
- **증상:** 피드에서 친구 게시물의 닉네임이 "이름없음"으로 출력
- **원인:** `users` 정책이 `for all using (auth.uid() = id)` — select도 본인만 가능. 다른 유저의 nickname을 join 으로 못 가져옴
- **해결:** `shares_group_with(target_user)` SECURITY DEFINER 함수로 "같은 그룹 멤버 여부" 체크 → select 정책 분리
- **배운 것:** RLS 설계 시 "본인만 수정"과 "본인 + 친구 읽기"를 분리해야 한다. `for all`은 너무 넓음. 작업별(`for select / update / insert / delete`)로 쪼개는 습관 필요
- **파일:** `supabase/migrations/0010_user_profile_visibility.sql`

### 5.5 RLS 닭과 달걀 — 초대 코드 가입 흐름
- **증상:** 시크릿 창의 두 번째 계정이 초대 코드 입력해도 "그룹을 찾을 수 없어요"
- **원인:** `groups` RLS가 "멤버만 select 가능". 가입 흐름은 "코드로 그룹 찾기 → 멤버 추가" 인데, 1단계에서 아직 멤버가 아니라 select 차단됨
- **해결:** `SECURITY DEFINER` RPC 함수 `join_group_by_code(code)` 만들어 RLS 우회. 함수 안에서 코드 검증 → 멤버 추가까지 원자적으로 처리
- **배운 것:** "초대로 진입하는 흐름"은 RLS 정책 설계 시 별도 고려 필요. RPC 함수가 권한 격리에 효과적
- **파일:** `supabase/migrations/0006_join_by_code.sql`

### 5.4b AI 한국어 자연스러움 (프롬프트 v2)
- **증상:** Haiku가 가끔 `"나랑 생겼다"` 같은 직역체 어색 표현 생성
- **원인:** 작은 모델은 한국어 자연스러움 약함. 긍정 예시만 있을 때 어색한 표현 가능성
- **해결:** 시스템 프롬프트에 "한국어 모국어 화자처럼" 명시 + **나쁜 예시(do-not) 섹션 추가**. `"× 나랑 생겼다 (→ 나랑 닮았다)"` 처럼 잘못된 표현과 교정안 함께 제공
- **배운 것:** Few-shot에서 긍정 예시뿐 아니라 **부정 예시 + 교정**이 효과적. 작은 모델은 명시적 가드레일이 필요함

### 5.4 AI 출력 형식 오염 (프롬프트 엔지니어링)
- **증상:** Haiku가 코멘트 앞에 `# 콩이의 한마디 🎧` 같은 마크다운 헤더 자동 삽입
- **원인:** 단일 user 메시지로 캐릭터 지시·형식 지시·요청을 다 섞어 보냄 → 모델이 "친절한 답변" 형태로 헤더까지 추가
- **해결:**
  1. **시스템 프롬프트 분리** — 캐릭터 정의·규칙은 system 메시지로 이동
  2. **금지 사항 명시** — "헤더, 라벨, 따옴표 절대 금지" 명시
  3. **Few-shot 예시 3개** — 노을·커피·강아지 예시로 출력 형식 학습
- **결과:** "곰돌이가 안경까지 쓰고 멋을 부리네, 나도 저렇게 멋있어 보고 싶은데 😎" — 깔끔
- **배운 것:** Haiku는 빠르고 싸지만 형식 통제는 Sonnet보다 약함 → Few-shot으로 명시적 학습 시켜야 함

### 5.8 토스 웹뷰 -1004 — 외부 OAuth 차단
- **증상:** 토스 앱에서 미니앱 테스트 시 로그인 버튼 누르면 `NSURLErrorDomain Error -1004 / 서버에 연결할 수 없습니다` 전체 페이지 에러
- **원인:** 토스 인앱 웹뷰는 외부 사이트(accounts.google.com)로의 페이지 이동을 차단. 구글 OAuth 리다이렉트가 막힘. 심사자도 동일하게 막혀 반려 확실
- **해결:** 외부 이동이 없는 토스 네이티브 식별 API `getAnonymousKey()`로 전환. 익명키 → Supabase Edge Function이 결정적(email/password) 계정으로 매핑 + `auth.admin.createUser`(멱등) → `signInWithPassword`로 세션 발급 → 프론트 `setSession`. 기존 RLS·스키마·`handle_new_user` 트리거 무수정 재사용
- **부가 처리:** 토스 딥링크(`intoss://harukong/{screenName}`)가 미정의 경로로 들어와 빈 화면 뜨는 것 방지 → React Router `path="*"` → 홈 폴백 추가
- **배운 것:** 임베디드 웹뷰는 일반 브라우저와 네트워크/네비게이션 정책이 다름. 플랫폼이 제공하는 1급 API를 우선 검토해야 함. 인증을 갈아끼우되 RLS 경계를 안 건드리도록 Edge Function을 어댑터로 둔 게 변경 범위를 최소화한 핵심

### 5.9 콩이 톤 어색함 — "사용자 기분"과 "사진 톤"의 불일치
- **증상(실사용 피드백):** 글 "비 와서 나가기 싫은 날,," + 강아지 사진 → 콩이가 강아지만 신나게 묘사("하얀 솜뭉치가 뭐 하는 거야 ㅋㅋ ... 마음이 절로 녹네, 너도"). 사용자 기분과 톤이 어긋나고 마지막 어미("너도")도 끊긴 느낌
- **원인:** Edge Function이 사진만 모델에 보내고 사용자 글은 무시 → 모델은 사진에만 반응 / Haiku는 1~2문장 안에서 두 흐름을 매끄럽게 잇지 못함
- **해결 (구조적):**
  1. **입력 확장:** `generateComment(base64, mediaType, userText)`로 글까지 전달, 서버에서 `사용자가 적은 한 줄 글: "..."` 형태로 user 메시지에 함께 주입
  2. **프롬프트 규칙:** "글이 있으면 글의 정서를 먼저 받고, 사진은 디테일로" + "문장 어미 반드시 완결" 명시
  3. **모델 상향:** Haiku 4.5 → Sonnet 4.5 (4.2 참고)
- **결과:** 같은 입력 → "비 오는 날엔 이 친구 옆이 최고지 🐶 같이 뒹굴며 푹 쉬어". 정서 일치 + 사진 짚기 + 응원 마무리 + 어미 완결
- **배운 것:** 멀티모달 앱은 "보이는 것"만 모델에 주면 사용자 의도를 절반밖에 못 잡음. 사용자가 직접 적은 텍스트는 가장 강한 의도 신호 — 무조건 같이 보내야 함

### 5.10 광고 API 함수명 오추정 — 버튼 눌려도 아무 반응 없음
- **증상:** "🎲 다른 한마디 듣기" 버튼이 화면에 보이지만 눌러도 아무것도 안 됨
- **원인:** `showRewardedAd`라는 함수가 `@apps-in-toss/web-framework`에 없음. 사용 가능한 함수를 추정해 코딩 → import 성공처럼 보이지만 undefined 호출 → catch 진입 → DEV 환경도 아니라 `rewarded=false` 유지 → 조용히 리턴. 에러 메시지도 표면에 안 뜸
- **발견 방법:** 실제 패키지 exports 목록 확인 (`node -e "console.log(Object.keys(require(...)))"`)
- **실제 API:** `loadFullScreenAd(options, onEvent, onError)` → `'loaded'` 이벤트 → `showFullScreenAd(options, onEvent, onError)` → `'userEarnedReward'` 이벤트. 두 함수 모두 콜백 기반이며 cleanup 함수를 리턴
- **해결:** Promise로 콜백을 감싸 async/await 구조로 직렬 처리. DEV 환경에선 광고 없이 바로 재생성
- **배운 것:** SDK 함수는 문서가 없으면 반드시 실제 exports·타입 정의 확인 후 코딩. "조용한 실패(silent failure)"는 분기 끝에 아무 효과 없이 return되는 구조에서 자주 발생 — 에러가 안 터지는 게 더 찾기 어렵다

## 9. 면접 예상 질문 + 답변 템플릿

### Q1. "AI를 어떤 식으로 통합했나요?"
A. Claude Vision API에 사진 + 사용자 한 줄 글을 함께 보내, 캐릭터 톤으로 1~2문장 코멘트를 받아 게시물에 저장합니다. **Supabase Edge Function(`claude-proxy`)을 거쳐 호출** — API 키는 서버 secret에만 두고, 게이트웨이에서 Supabase JWT를 검증해 로그인한 사용자만 호출하게 막았습니다. 시스템 프롬프트 자체도 서버에 둬서 클라이언트 번들에서 안 보입니다. 사진 업로드와 AI 호출은 `Promise.all`로 병렬 처리해 대기를 줄였고, AI 실패 시에도 게시물 저장은 정상 완료(graceful degradation)되게 했습니다.

### Q2. "비용은 어떻게 관리하나요?"
A. 컨셉상 **1인 1일 1회 호출**이 자연스러운 상한이라 트래픽 자체가 제한적입니다. v1.0~v1.1은 Haiku 4.5(회당 ~5원), v1.2는 글+사진 통합 입력 도입하며 Sonnet 4.5(회당 ~14원)로 상향했습니다. 100명 매일 사용 기준 월 ~4만 원으로 부담이 적고, 회당 비용보다 **품질 차이가 캐릭터형 앱의 차별화에 더 중요**하다고 판단했습니다. 사용자 폭증 시 Haiku 다운그레이드 / 인앱 광고 도입을 옵션으로 두고 있습니다.

### Q3. "Supabase RLS 어떻게 설계했어요?"
A. 5테이블에 각각 RLS 켜고 그룹 멤버 기반 격리 정책을 적용했습니다. 다만 group_members 자기 참조 시 무한 재귀가 발생해, `SECURITY DEFINER` 함수로 우회했습니다.

### Q4. "토스 로그인 연동에서 어떤 문제를 겪었고 어떻게 풀었어요?"
A. 개발은 구글 OAuth로 빠르게 하고 출시 직전 교체할 계획이었는데, 토스 인앱 웹뷰가 외부 사이트 페이지 이동을 막아 구글 OAuth가 `-1004`로 실패했습니다(심사자도 로그인 불가). 정석인 `appLogin`(OAuth)은 mTLS 인증서 + 토스 Client ID 메일 발급(수일)이 필요해 마감 리스크가 컸습니다. 그래서 토스가 공식 제공하는 `getAnonymousKey()`로 사용자를 식별하고, Supabase Edge Function에서 그 키를 결정적 계정으로 매핑해 세션을 발급하는 구조로 전환했습니다. **기존 RLS(`auth.uid()`)를 한 줄도 안 고치고** 인증만 갈아끼운 게 핵심입니다. 위험도(친구초대 기반 저위험)에 맞춰 과한 서버검증 대신 적정 수준을 택한 판단이기도 합니다.

### Q5. "혼자 다 하셨어요?"
A. 기획·DB 설계·프론트·인증·AI 통합·배포·운영 1인 풀사이클입니다. AI 페어 프로그래밍(Claude Code) 적극 활용해 생산성 끌어올렸고, 의사결정·트레이드오프 판단은 직접 했습니다.

### Q8. "스트릭 같은 사용자 통계는 어디서 계산했나요?"
A. **DB 트리거**로 백엔드에서 처리합니다. 게시물 insert 시 `update_streak_on_post()` 함수가 자동 실행되어 users 테이블의 `streak`/`last_posted_at`을 갱신합니다. 클라이언트 계산은 시간 조작 가능성으로 신뢰성이 떨어지므로, 시간 기반 게이미피케이션은 무조건 서버 사이드 처리해야 합니다.

### Q7. "이모지 반응 같은 인터랙션 UX 어떻게 처리하셨어요?"
A. **낙관적 업데이트(Optimistic Update)** 적용. 사용자가 이모지 누르는 즉시 화면을 갱신하고, DB 호출은 백그라운드에서 처리합니다. 임시 ID로 일단 반영했다가 DB 응답이 오면 진짜 ID로 교체합니다. 네트워크 지연이 있어도 클릭 → 반응이 즉시 보여 사용감이 부드럽습니다.

### Q6. "프롬프트 엔지니어링은 어떻게 하셨어요?"
A. 처음엔 단일 user 메시지에 캐릭터·규칙·요청을 다 섞었더니 모델이 마크다운 헤더(`# 콩이의 한마디`)를 자동으로 붙였습니다. 이를 (1) **system 프롬프트로 캐릭터·규칙 분리**, (2) **금지 사항 명시**("헤더, 라벨 금지"), (3) **Few-shot 예시 3개**(노을·커피·강아지)로 해결했습니다. Haiku 같은 작은 모델일수록 명시적 예시가 효과적이라는 걸 배웠습니다.
